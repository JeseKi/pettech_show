输出目录固定为 `main/<output_date>/<output_id>/`，必须创建这些文件：

- `main.md`
- `metadata.json`

`output_date` 使用当天当前日期的 `YYMMDD` 格式，除非用户明确指定另一个发布/输出日期。不要从素材文件名推导输出日期。

`output_id` 从 `output_date` 和 `main/<output_date>/` 下当天文章顺序共同确定，不直接继承素材编号。每天第一篇必须从 `_1` 开始，然后按 `_2`、`_3` 顺序递增，不允许跳号：

- 如果今天是 2026-05-10，`output_date` 为 `260510`，且 `main/260510/260510_*` 不存在，第一篇应输出到 `main/260510/260510_1/`。
- 如果 `main/260510/260510_1/` 已存在，下一篇应输出到 `main/260510/260510_2/`。
- 如果 `main/260510/260510_1/` 和 `main/260510/260510_2/` 已存在，下一篇应输出到 `main/260510/260510_3/`。

如果一篇主稿使用多个素材，素材日期和素材自身的 `_1/_2/_5` 编号不参与文章目录命名。所有实际使用的素材仍记录进 `materials_used`。

示例：如果今天是 2026-05-10，即使主素材是 `material/260505/260505_5_Codex 额度不够怎么办？.json`，只要它是当天生成的第二篇主稿，目录也必须是 `main/260510/260510_2/`，不是 `main/260505/260505_5/`。

写作流程要求：

1. 先参考 `references/style_reference.md` 和本文档要求校准详略、节奏和成稿感。
2. 读取 `material/<date>/*.json`，判断哪些素材适合进入主稿。
3. 根据 `元数据.raw文件路径` 按需阅读 `raw/<date>/*.md`，补足事实细节、案例、步骤和推广信息。
4. 先做一个简短提纲，至少写清主线和 2 到 4 个正文小节。
5. 先创建对应输出目录，例如今天是 2026-05-10 时创建 `main/260510/260510_1/`。
6. 先完成 `main/<output_date>/<output_id>/main.md`。
7. 最后写 `main/<output_date>/<output_id>/metadata.json`。

各文件要求：

- `main.md`：默认公众号主稿，必须完整，也必须像正式成稿。正文顶部不要写大标题，标题写进 `metadata.json`。
- 文章长度由素材复杂度、信息密度和表达完整度决定。
- 写完整、写透、节奏自然；不要为凑字扩写，也不要因为旧规则压缩必要展开。
- 不要说废话。保留事实、判断、操作步骤、风险提醒和转化信息，删除重复铺垫、泛泛解释和空洞段落。
- 主稿正文必须是纯文本 Markdown，不允许携带任何图片。禁止输出 Markdown 图片语法、HTML `<img>`、图片占位符、封面图块、二维码图块，或用于渲染图片的链接。
- 原文中的截图、对比图、证明材料、二维码、操作示例和关键视觉证据只能转写为正文里的文字信息；不要保留、复制、嵌入、链接或占位这些图片。

`main.md` 推荐结构：

- 2 到 3 段导语，先讲“为什么今天值得看”，不要在正文里写大标题。
- 2 到 4 个正文小节，用代表性素材、痛点、热点或解决方案支撑主线。
- 一个简洁收束，给出今天的核心观察或行动方向。

每个正文小节尽量包含：

- 一个或多个具体信息点、用户痛点或方案变化。
- 对这些信息的解释，而不是只复述素材。
- 支撑该段判断的论据，不能只有观点没有展开。
- 对开发者、团队、使用者或行业的影响判断。
- 每段都要有明确作用；如果只是重复前文，就删掉。

正文不要默认写成编号清单。只有在某一段里需要快速列出重点时，才允许短列表。

`metadata.json` 必须采用下面的结构：

```json
{
  "input_mode": "filesystem",
  "output_id": "260510_1",
  "topic": "选题",
  "pain_point": "痛点",
  "solution": "解决方案",
  "hook": "钩子",
  "article": {
    "role": "main",
    "file": "main.md",
    "title": "标题",
    "summary": "摘要",
    "tags": ["产品介绍", "AI 编程"],
    "search_intents": [
      {
        "role": "primary",
        "意图类型": "痛点型",
        "关键词": "Claude Code 太贵",
        "搜索意图": "用户已经感受到价格或额度压力，想判断问题本质或替代方案。",
        "适合文章角度": "从价格焦虑切入，解释 AI 编程工具进入新产品阶段。",
        "标题使用建议": "建议完整保留",
        "优先级": "高",
        "来源依据": "来自 material 的搜索入口，本文正文可以承接该问题。"
      }
    ],
    "materials_used": [
      {
        "metadata_file": "material/260505/xxx.json",
        "raw_file": "raw/260505/xxx.md",
        "title": "素材标题"
      }
    ]
  }
}
```

`topic` 规则：

- 每篇文章都必须包含顶层 `topic` 字段。
- `topic` 应填写这篇文章的简短选题名，而不是标题、摘要或素材文件名。
- `topic` 应稳定、可检索，例如 `ChatGPT Plus账号安全检测`、`GPT聊天记录备份`、`Codex额度方案`。
- 如果用户提供的 CSV/规划表当前行有非空 `topic` 列，优先使用该值。

`pain_point`、`solution`、`hook` 规则：

- 每篇文章都必须包含顶层 `pain_point`、`solution`、`hook` 字段。
- 如果用户提供的 CSV/规划表当前行有对应列，原样写入对应值。
- 如果没有对应列或值为空，写空字符串。
- 这些字段用于后续内容生产和检索，不要把字段名写进 `main.md` 正文。

`tags` 规则：

- 每篇文章都必须包含 `tags`。
- 必打主标签必须二选一：`产品介绍` 或 `产品相关新闻`。
- 可以在主标签之外继续追加可选扩展标签。
- 不要同时打上两个主标签。

`search_intents` 规则：

- `article.search_intents` 用来记录这篇文章实际承接的搜索入口，供后续标题生成、变体分配和覆盖检查使用。
- 优先从已使用 material JSON 的 `搜索入口` 字段或 `wiki/search-intents/` 页面选择，不要凭空编造。
- 至少填写 1 个 `role: primary`，当素材足够时再填写 2 到 4 个 `role: secondary`。
- 每个条目保留中文字段：`意图类型`、`关键词`、`搜索意图`、`适合文章角度`、`标题使用建议`、`优先级`、`来源依据`。
- 只记录正文可以真实回答的入口；如果正文没有承接，不要为了 SEO 把关键词写进 metadata。
- 如果没有任何可用搜索入口，写空数组 `[]`，不要省略字段。

`materials_used` 规则：

- 只记录实际使用过的素材。
- `metadata_file` 填写 `material/<date>/` 下的 JSON 路径。
- `raw_file` 填写该素材实际阅读或引用过的原文路径；如果没有读取原文，可以填空字符串。
- `title` 填写素材标题，优先使用 `元数据.标题`。

文章风格要求：

- 像公众号成稿，不像信息摘抄或素材汇总。
- 先有观点，再有信息，不要只有信息没有主线。
- 先有观点，也要有论据；每个关键判断都应配套说明它为什么成立。
- 细节要够，不能只做趋势判断或抽象评论。
- 避免空泛感叹。
- 直入主题，少写大而泛的背景。
- 标题可以有传播感，但不能偏离正文事实。
- 不要写“下面我们来看看”“基于本地文件整理”等暴露写作过程的话。
- 不要写“有素材提到”“材料显示”“这篇素材指出”“根据 metadata/raw”等内部整理口吻；事实来源要转换成读者语言，例如“行业报道里提到”“公开案例里提到”“有从业者算过一笔账”“一位创作者的复盘里提到”“公开数据里能看到”。
- 写完 `main.md` 后必须做一次发布前词汇扫描：`素材`、`材料`、`metadata`、`raw`、`本地文件`、`提示词` 等词不能以写作过程含义出现在正文里。
- 展开方式以 `references/style_reference.md` 和本文件的结构要求为准，不要复刻固定样稿的句子或结构表面形式。
- 不需要为论据附出处链接或引用格式，但需要把支撑判断的事实依据直接写进正文。
